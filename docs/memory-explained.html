🎮 Nuke Raider — How the ROM's Memory Works

An explain-it-like-I'm-five tour of Game Boy memory: cartridge banks, VRAM, OAM and WRAM. Click any colored box — the panel at the bottom tells you what it is.

The 30-second version. A Game Boy is a tiny computer with almost no memory. The big game lives on the cartridge as ROM (read-only, can't change). But the CPU can only "see" a small window of it at a time, so the cartridge swaps chunks in and out — those chunks are called banks. The console's own scratch memory is split into special-purpose rooms: VRAM (the picture book of tiles & colors), OAM (the list of moving sprites), and WRAM (variables & game state). Everything below is real, measured from this build of the ROM.

③The cartridge banks (why we don't run out of room)

The swappable window (orange stripe above) is only 16 KB. But the cart holds 512 KB. So the game is sliced into 32 banks of 16 KB each, and the cart swaps one into the window when needed. Bank 0 is special — it's always visible, so the most important code lives there. Click any bank.

④VRAM — the picture book (tiles)

The Game Boy can't draw free-form pixels. It draws 8×8 tiles — like LEGO bricks. VRAM stores up to 384 tile pictures (192 per color bank × 2 on the GBC). Backgrounds and the HUD all share one strip. Click a section.

⑤OAM — the moving things (per scene!)

A sprite is a tile that moves freely (your car, bullets, enemies). The hardware allows exactly 40 sprite slots — no more, ever. But here's the trick: every scene reuses the same pool. When the game changes scene it wipes the sprite list and refills it, so only one scene's sprites are ever on screen. The budget that matters is the busiest single scene — not the whole game added up.

Myth-buster: the old "35/40" was a build-time worst-case sum. Real OAM is per-scene, and the only scene that gets close to the wall is Playing (32/40). Pick a scene below and watch the 40 slots fill differently.
